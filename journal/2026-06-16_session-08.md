## Decisions made

| Area | Decision | Reasoning |
|---|---|---|
| Civ knowledge seeding | **Manual JSON files** — reversed the session 01 decision (was "organic, no manual seed") | Organic seeding can't provide civ-specific strategic context the agent needs before it has seen many games; reflection kept failing `historical_context_present` because the agent had no domain knowledge source |
| Knowledge file format | JSON, one file per civ | Consistent schema per civ; structured enough for uniform ingestion; one file per civ keeps git diffs clean and allows updating a single civ without touching others |
| Knowledge schema | `civ`, `strengths`, `weaknesses`, `unique_unit`, `unique_tech` (nested by age), `economy_bonuses`, `team_bonus`, `build_order_notes`, `common_pairings`, `power_spikes` (nested by age) | Schema emerged from the Britons file the user filled in; `unique_tech` and `power_spikes` as nested objects by age are richer than flat strings |
| `_template.json` convention | Underscore prefix, stored in the same folder as civ files | Keeps template co-located with data; underscore prefix sorts it to the top and makes it visually distinct from real civ files |
| Prompt tuner promotion | Manual only — this was already in the code but the BLUEPRINT described it as automatic | Auto-promotion risks a bad run batch silently overwriting a working prompt; the docstring in `prompt_tuner.py` already stated the reason; BLUEPRINT was wrong, not the code |
| BLUEPRINT update | Added `ingestion.py` to memory/ section, `data/knowledge/` to data section, fixed `prompt_tuner.py` description, added "The human improvement loop" section | Blueprint was missing the human participant in the improvement loop entirely; this was a conceptual gap, not just missing documentation |
| CLAUDE.md update | Added Documentation section requiring README to always describe the human improvement workflow | Prevents future sessions from shipping improvements to the loop without documenting how to use them |

## Things decided against

- **Auto-promotion in `prompt_tuner.py`** — would require the agent to edit its own prompt file; too much risk of silent regression from a noisy run batch
- **Single `civs.json` file** — one file per civ is better for partial updates and readable diffs
- **Markdown for civ knowledge** — JSON is more structured and can be validated; markdown is easier to write but harder to parse consistently in ingestion

## What was built

- `memory/ingestion.py` — reads `data/knowledge/civs/*.json`, skips `_template.json`, converts each civ to a prose document, upserts into ChromaDB `civ_knowledge` collection; idempotent
- `data/knowledge/civs/_template.json` — schema template for future civ files
- `data/knowledge/civs/britons.json`, `persians.json`, `khmer.json`, `byzantines.json` — filled in by the user with full civ data
- `KNOWLEDGE_PATH` added to `config.py` and `.env.example`
- `__main__` block added to `learning/prompt_tuner.py` — now runnable via `python -m learning.prompt_tuner`
- "How to improve the agent" section added to `README.md` — 6-step workflow covering reflection, prompt editing, prompt tuner, ingestion, and evaluator updates
- BLUEPRINT.md updated with 4 corrections and 1 new section
- CLAUDE.md updated with Documentation rule
