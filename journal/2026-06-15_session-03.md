# Session 03 — 2026-06-15

## What was discussed

Resumed from the end of session 02. The entire v1.1 codebase was written but had never
run end-to-end. The primary blocker was `parse_replay` being stubbed with
`NotImplementedError` because mgz 1.8.51 could not parse replays from patch

## Decisions made

### mgz fork pin — Kjir/aoc-mgz commit b4a30d8

PyPI mgz 1.8.51 and the git master HEAD both fail on the replay file
(`MP Replay v101.103.47452.0 @2026.06.13 143618 (1).aoe2record`). The git master has
the same version tag as PyPI — the Last Chieftains fixes (PR #142, merged Mar 3 2026)
appear to be on a branch not yet tagged for release.

The `Kjir/aoc-mgz` fork (commit b4a30d8) parses the file successfully. It is pinned in
`requirements.txt` with a migration note back to the official package.

**Migration plan (in requirements.txt):** once `happyleavesaoc/aoc-mgz` releases a
version that passes a round-trip parse on a current-patch replay, replace the git pin
with `mgz>=<new_version>`.

### parse_replay implementation — use `mgz.model.parse_match()`

`mgz.summary.Summary` returns civilisation as an integer. `mgz.model.parse_match()`
returns it as a string (`'Britons'`, `'Aztecs'`, etc.), matching what the SPEC requires
(no `civ_NN` fallbacks in output).

`parse_match()` also exposes: player name, civ string + id, eAPM, winner bool,
`profile_id` (for AI detection via `0xFFFFFFFF` sentinel), team lists, map name,
duration (timedelta), game type, completed flag.

AI detection confirmed working: AI players have `profile_id == 4294967295 == 0xFFFFFFFF`
and empty names.

### Timeout check position — only block on fresh turns

The first end-to-end run completed all meaningful work (report written, run entry
written, ChromaDB memory stored) but the 120-second timeout fired on the final loop
iteration — the one where we would have sent back the tool results and received Claude's
closing text, which would have triggered the evaluator and `log_run`.

Root cause: the `thinking={"type": "adaptive"}` call took ~2m45s to generate the
coaching report. By the time tool results came back, >120s had elapsed, and the timeout
check at the top of the next iteration fired before the final API call.

Fix: added a `pending_tool_results` flag in `core/agent.py`. The timeout is only checked
when there are no pending tool results to dispatch — i.e., before a fresh "speculation"
turn, not before responding to a tool-use response Claude already returned.

## Things explicitly decided against

- **Increasing the timeout to 300s**: the user chose the structural fix over a number
  change, so SPEC.md §4's 120s limit stands. The fix allows the run to complete fully
  even when the API call itself takes longer than 120s.
- **Disabling adaptive thinking**: kept. The first run's report quality was high and
  justified the latency.
- **AoEInsights/mgz-fast fork**: this installs as `mgz.fast` (a C extension namespace),
  not a drop-in replacement for `mgz.summary` or `mgz.model`. Not usable.
