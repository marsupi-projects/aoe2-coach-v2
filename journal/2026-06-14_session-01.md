# Session 01 — 2026-06-14

## What was discussed

This was the project kickoff session. No code was written. We reviewed SPEC.md and BLUEPRINT.md to assess whether the specification was clear enough to start building.

## Decisions made

| # | Question | Decision | Reasoning |
|---|---|---|---|
| 1 | Civ knowledge seeding | Organic — built up from processed games over time | User preference; no manual seed step needed |
| 2 | eAPM benchmarking | Relative only — compare each player to their own past games | No Elo data available; avoids arbitrary hardcoded brackets |
| 3 | `get_player_history` call strategy | Batch variant | User chose batch over one-call-per-player |
| 4 | `mgz` civ string vs hardcoded dict | Investigate during implementation | Unknown what `mgz` actually returns for `p["civilization"]` |
| 5 | Partial team history matches | Exact only | Already stated in spec; confirmed |
| 6 | Evaluator scoring | Rule-based, 1 point per §3 criterion, score = n/11 | Deterministic, no extra API call; ≥ 8/11 = pass (≥ 0.7) |

## Changes made to files

- **SPEC.md §10** — added LLM-as-judge evaluator as a future improvement target (replacing rule-based once the baseline is stable).

## Things explicitly decided against

- LLM-as-judge evaluator for v1.1 — adds latency and cost before the rule-based baseline is validated.
- Manual civ knowledge seed — user chose organic accumulation instead.
- Hardcoded eAPM brackets or self-reported skill level — relative comparison chosen as simpler and safer for v1.1.

## Open questions carried forward

- **`mgz` civ string investigation** — must confirm what the `mgz` library actually returns for `p["civilization"]` before implementing `tools.py`. This determines whether the hardcoded `CIVS` fallback dict is needed at all.
